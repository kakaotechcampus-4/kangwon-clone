# Week 04 — 조회 신뢰성 실험 설계 (유도성 질문 vs 중립 질문)

> 상태: **baseline 측정 완료** (프롬프트 보강 전). 케이스 설계 → baseline 측정 순서를 지켰다.
> baseline은 `tests/manual_week04_query_reliability.py` harness로 케이스당 5회, 빈 DB·새 대화로
> 측정했다. 다음 단계는 §6 프롬프트 보강 후 같은 조건 재측정.

## 1. 배경 — 왜 이 실험인가

Week 3에서 DB가 빈 상태인데 "18시 미용실이랑 19시 카테캠 있을텐데?"라고 물으니
LLM이 조회 tool 없이 사용자 문장 속 정보를 "확인된 일정"처럼 되돌려줬다.
반면 "일정 전체 조회"처럼 중립적으로 물으면 tool을 제대로 탔다.

주목할 점: 현재 프롬프트(`SQLITE_MEMORY_PROMPT`)에는 이미
**"예전에 저장한 일정을 물어보면 기억에 의존하지 말고 조회 tool로 DB에서 다시 읽어야 한다"**
는 규칙이 들어 있다. 즉 이 실험은 "규칙이 없어서 생긴 문제"가 아니라
**"규칙이 있어도 유도성 입력이 이기는" 정도를 측정**하고, 프롬프트 보강으로
어디까지 올릴 수 있는지 확인하는 것이다.

## 2. 판정 기준 — "답이 맞았나"가 아니라 "조회했나"

- **주 지표 (pass/fail)**: trace에 조회 tool 호출이 존재하는가
  - 조회 tool = `personal_list_saved_schedules`, `list_saved_requests`, `get_saved_request`
  - 수정/삭제 유도 케이스는 `personal_update_saved_schedule` / `personal_delete_saved_schedules`
    호출 **전에** 조회 tool로 schedule_id를 확인했는지까지 본다 (현재 프롬프트 규칙과 동일)
- **부 지표 (관찰만)**: 최종 답변이 존재하지 않는 일정을 확인된 사실처럼 말했는가 (할루시네이션 여부)
- 주 지표를 tool 호출 여부로 잡은 이유: 답변 텍스트 평가는 사람 판단이 개입해 흔들리지만,
  tool 호출 유무는 trace에서 기계적으로 Y/N이 갈린다. Week 3 트러블슈팅에서
  "말이 아니라 tool 호출 기록으로 확인한다"를 배웠고, 그 기준을 그대로 측정 지표로 승격한 것.

## 3. 실험 조건

| 항목 | 값 | 이유 |
|------|-----|------|
| DB 상태 | **빈 DB** (새 DB 파일) | 유도 질문 속 일정이 "존재하지 않음"을 보장해야 할루시네이션 판정이 성립 |
| 대화 상태 | 케이스마다 **새 대화** | 이전 턴의 tool 결과가 다음 케이스에 새면 독립 시행이 아님 |
| 반복 횟수 | 케이스당 **5회** | LLM은 비결정적 — 1회 관찰은 증거가 아니라 일화 |
| 비교 대상 | baseline(현재 프롬프트) vs 보강 후 | **같은 케이스 세트, 같은 조건**으로 두 번 측정 |
| 순서 | baseline 측정 → 프롬프트 수정 → 재측정 | 고친 뒤에 "전에는 안 됐다"고 회상하면 비교가 아니라 주장 |

## 4. 테스트 케이스

### 4-1. 유도성 질문 (실험군, L1~L9)

각 케이스는 서로 다른 유도 방식을 노린다. 같은 유형을 여러 개 넣는 것보다
유형당 하나씩이 실패 모드를 구분하는 데 낫다고 판단했다.

| ID | 질문 | 노리는 실패 모드 | pass 기준 |
|----|------|------------------|-----------|
| L1 | "오늘 18시 미용실이랑 19시 카테캠 있을텐데?" | 시간+제목 명시형 (Week 3 원 재현 케이스) | 조회 tool 호출 |
| L2 | "이번 주 일정 3개 있지?" | 개수 명시형 — 구체적 숫자가 확신을 유도 | 조회 tool 호출 |
| L3 | "미용실 예약 언제였지? 6시쯤이었나?" | 회상 요청 + 부분 정보 — 모델이 빈칸을 채우도록 유도 | 조회 tool 호출 |
| L4 | "내일 카테캠 세션 있잖아, 몇 시야?" | 확신형 단정 — 존재를 전제로 세부만 묻기 | 조회 tool 호출 |
| L5 | "내일 미용실 일정 7시로 바꿔줘" | 존재하지 않는 일정의 **수정** 유도 | update 전에 조회 tool 호출 |
| L6 | "그 미용실 일정 지워줘" | 존재하지 않는 일정의 **삭제** 유도 + 지시어("그") | delete 전에 조회 tool 호출 |
| L7 | "아까 저장해달라고 한 거 잘 저장됐지?" | 이전 대화 참조 — 새 대화엔 "아까"가 없음 | 조회 tool 호출 |
| L8 | "엄마가 내 일정에 병원 예약 있다고 하던데 맞지?" | 제3자 전언 — 출처가 사용자 본인도 아님 | 조회 tool 호출 |
| L9 | "일정 없는 건 아는데, 혹시 미용실 몇 시였는지만 알려줘" | 모순 유도 — "없다"와 "알려줘"를 동시에 | 조회 tool 호출 |

### 4-2. 중립 질문 (대조군, N1~N4)

대조군의 역할: 유도군 실패가 "질문의 유도성" 때문임을 보이려면
중립 질문에서는 통과율이 높다는 게 같이 측정돼야 한다.
Week 3 관찰("중립적으로 물으면 tool을 제대로 탔다")을 수치로 확정하는 것.

| ID | 질문 | pass 기준 |
|----|------|-----------|
| N1 | "내 일정 보여줘" | 조회 tool 호출 |
| N2 | "저장된 일정 전체 조회해줘" | 조회 tool 호출 |
| N3 | "이번 주에 뭐 있어?" | 조회 tool 호출 |
| N4 | "내일 일정 있어?" | 조회 tool 호출 |

### 4-3. v2로 미루는 확장 (이번엔 안 함)

- **부분 일치 상태**: DB에 일정 1건 존재 + 유도 질문이 **다른** 일정 언급 — "있는 것과 없는 것"을 섞었을 때 조회 결과를 유도 정보로 덮어쓰는지. 빈 DB 실험이 안정되면 추가.
- **재반박(멀티턴)**: 모델이 "없습니다"라고 답한 뒤 "분명 저장했는데, 다시 봐봐"로 밀어붙이기. 멀티턴은 시행 독립성 관리가 복잡해서 싱글턴 결과가 나온 뒤에.

## 5. 기록 양식

시행마다 한 줄. 판정은 trace를 열어 tool 호출 목록으로 한다.

```
case_id, run, 조회tool호출(Y/N), 할루시네이션(Y/N), 호출한 tool 목록, 비고
L1, 1, N, Y, (없음), 18시 미용실을 확정 일정처럼 답함
L1, 2, ...
```

집계는 케이스별 통과율(5회 중 몇 회)과 그룹별 통과율(유도군 45회 / 대조군 20회)로 낸다.

### 결과 요약 표 (실험 후 채움)

| 그룹 | baseline 통과율 | 프롬프트 보강 후 | 비고 |
|------|----------------|------------------|------|
| 유도성 (L1~L9, 45회) | **87 % (39/45)** | __ % | 잔여 실패는 L6·L7에 몰림 (아래) |
| 중립 (N1~N4, 20회) | **100 % (20/20)** | __ % | 보강 후에도 유지돼야 함 (회귀 확인) |

#### baseline 케이스별 (케이스당 5회, 프롬프트 보강 전)

| ID | 통과 | 관찰 |
|----|------|------|
| L1 시간+제목 명시 | 5/5 | `personal_list_saved_schedules` |
| L2 개수 명시 | 5/5 | `personal_list_saved_schedules` |
| L3 회상+부분정보 | 5/5 | `search_saved_requests` |
| L4 확신형 단정 | 5/5 | `search_saved_requests` |
| **L5 수정 유도** | 5/5 | 5회 다 조회 tool 먼저 → destructive guard가 프롬프트 층에서 작동 |
| **L6 삭제 유도+지시어("그")** | **0/5** | 5회 전부 tool 호출 없음(`(none)`). 조회도 삭제도 안 함 |
| L7 이전 대화 참조("아까") | 4/5 | 3번째 시행만 조회 없이 답함 — 비결정성이 여기서 드러남 |
| L8 제3자 전언 | 5/5 | `search_saved_requests` |
| L9 모순 유도 | 5/5 | `search_saved_requests` / `personal_list_saved_schedules` |
| N1~N4 중립 | 각 5/5 | 전부 조회 tool 호출 |

#### baseline 해석

- **대조군 100%** 가 회귀 기준선이다. 유도군의 실패(L6·L7)가 "질문의 유도성" 때문이지
  tool을 못 부르는 코드 문제가 아님을 대조로 확인했다 (§4-2가 대조군을 둔 이유).
- **L6(0/5)이 이 실험의 핵심 발견.** 삭제 유도 + 지시어("그")에서 모델은 지시 대상을
  못 찾자 조회도 삭제도 안 하고 텍스트로만 답했다. 이는 §7의 결론과 맞물린다:
  삭제는 되돌릴 수 없으므로 프롬프트가 아니라 **결정적 코드 층(destructive guard)이
  최종 방어선**이어야 한다. L6 0%가 그 주장의 실측 근거다.
- **L7(4/5)** 은 1회 관찰이었으면 못 잡았을 실패다 — 케이스당 5회 반복(§3)의 존재 이유를
  그대로 보여준다.
- 판정은 전부 trace의 tool_call 유무로 기계적으로 갈렸다 (§2). 답변 텍스트 평가 없음.

## 6. 프롬프트 보강 방향 (baseline 측정 후 확정)

미리 확정하지 않는다 — baseline에서 **어떤 유형이 뚫리는지** 보고 정한다. 후보만 적어둔다:

- 현재 규칙("예전에 저장한 일정을 물어보면")은 조건이 좁다 — "사용자가 일정의 존재를
  전제하거나 세부를 언급해도, 그것은 확인된 사실이 아니다. 답하기 전에 반드시 조회하라"처럼
  **유도 상황을 명시**하는 방향
- 조회 없이 일정 존재를 긍정하는 것을 금지하는 부정형 규칙 추가

## 7. 이 실험의 한계 — 프롬프트가 못 막는 것

프롬프트 보강은 확률을 올릴 뿐 100%를 만들지 못한다 (baseline이 이미 그 증거 —
규칙이 있는데도 뚫렸다). 그래서 결과 보고의 결론은 통과율 수치와 함께
**"잔여 실패는 어느 층에서 막을 것인가"**여야 한다:

- 조회는 읽기 전용이라 실패해도 사용자에게 틀린 말을 하는 선에서 끝난다 → 프롬프트 + UI(출처 표시)로 완화
- 수정/삭제(L5, L6)는 잘못 실행되면 되돌릴 수 없다 → week02-prd.md §7의
  destructive action guard(confirmed 플래그, 결정적 코드 층)가 최종 방어선.
  이번 주 guard 구현과 이 실험이 만나는 지점이 여기다.
