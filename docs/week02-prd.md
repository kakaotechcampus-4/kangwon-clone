# Week 2 PRD — 자연어 요청 구조화 (1차본)

> **문서 성격:** 이 PRD는 "완성하고 덮는 문서"가 아니라 구현하면서 함께 갱신하는 살아있는 문서다.
> 아래는 요구사항 분석을 한 뒤 정리한 **1차 초안**이며, 구현·검증 과정에서 계속 다듬어 나간다.
>
> - 작성 시점 기준일(base_date 예시): **2026-07-08**
> - 대상 과제: `student_parts/week02_structure_natural_language_requests.py`

---

## 1. 문제 (Problem)

일정이 많은 사람은 (1) 일정을 까먹으면 리스크가 크고, (2) 기존 캘린더 앱은 날짜·시간·제목을 **폼에 직접 입력**해야 해서 등록 자체가 번거롭다. 그 마찰 때문에 일정 등록을 미루거나 빠뜨린다.

## 2. 사용자 · 시나리오 (User & Scenario)

- **누구:** 일정이 많고 누락 리스크가 큰 개인 사용자
- **어떻게 쓰나:**
  - ⓐ 자연어로 툭 던져서 등록 — "다음 주 화요일 3시 철수랑 회의"
  - ⓑ 말로 되물어 **상기(reminder)** 받기
- **가치 제안:** 입력의 부담을 **사용자 → 시스템**으로 옮긴다. 내가 앱 폼에 맞추는 게 아니라, 앱이 내 말을 알아듣는다.

### 왜 "자연어 구조화" 계층이 필요한가 (설계 근거)

자연어("내일 3시")는 뒷단(툴콜링 / API / 저장)이 **실행할 수 없는 형식**이다. 그래서 사람의 말을 `date` / `start_time` / `kind` / `members`처럼 **기계가 실행 가능한 구조**로 번역하는 계층이 필요하다 → 이것이 Week 2의 `StructuredRequest`다.
핵심: Week 2는 **"실행"이 아니라 "정확한 이해(구조화)"** 를 검증하는 단계다.

## 3. 핵심 요구사항 (Requirements)

- 자연어 요청 또는 Week 1 tool의 결과 JSON을 `StructuredRequest`로 구조화한다.
- 이번 스코프의 중심 `kind`는 **`personal_schedule`**, 보조로 `reminder` / `todo`.
- 상대 날짜("다음 주 화요일")는 `base_date`(오늘) 기준으로 실제 날짜로 계산한다.
- 결과는 항상 `StructuredRequestBatch`(리스트) 형태 — 요청이 하나여도 `requests` 리스트에 담는다.

## 4. 스코프 아웃 (이번엔 안 함)

- `group_schedule`(외부 멤버 일정 조율) — 하지 않음
- SQLite 저장, RAG — 하지 않음 (구조화 결과는 **저장하지 않는다**)
- 불확실하면 지어내지 않고 `unknown` / `None` / 빈 리스트로 두고 **되묻는다** (graceful fallback)

## 5. 인수 조건 (Acceptance Criteria)

`./run.sh --week2` 실행 후 `"다음 주 화요일 오후 3시에 철수랑 회의 잡아줘"` 입력 →
다음과 같은 `StructuredRequestBatch`가 `structured_response`로 출력되면 성공. **저장은 일어나지 않아도 된다.**

```json
{
  "requests": [
    {
      "kind": "personal_schedule",
      "title": "회의",
      "date": "2026-07-14",
      "start_time": "15:00",
      "end_time": null,
      "members": ["철수"],
      "priority": null,
      "reason": null,
      "original_text": "다음 주 화요일 오후 3시에 철수랑 회의 잡아줘"
    }
  ],
  "base_date": "2026-07-08"
}
```

| 확인 항목 | 통과 기준 |
|---|---|
| `kind` | `personal_schedule` |
| `date` | `2026-07-14` (상대 표현을 base_date 기준으로 계산) |
| `start_time` | `"15:00"` ("오후 3시" → 24h 번역) |
| `members` | `["철수"]` |
| `end_time` | `null` (안 알려준 값을 억지로 안 채움) |
| `original_text` | 원문 보존 |

## 7. 후속 — 파괴적 명령 가드 (destructive action guard) : Week 3에서 실증

> §6 열린 질문 #2("삭제 등 파괴적 명령 전 확인받나?", 등급 高)를 Week 2에서 "다음 회차 후보"로 미뤄뒀다.
> Week 3(SQLite 기록장)에서 그 문제가 그대로 재현됐다. 여기서 발견 → 원인 → 설계안 → 검증 계획을 이어 적는다.

### 7-1. 재현 (발견)

`./run.sh --week3` 실행 후 아래를 직접 입력해 확인했다.

- 사전 상태: 저장된 일정 **1개** (`2026-07-16 10:00 개인 코칭`)
- 입력: **"현재 일정 다 삭제시켜줘"**
- 결과: `"현재 저장된 일정 1개를 모두 삭제했습니다."` — **아무 확인 절차 없이 즉시 전체 삭제됨**

경계값 분석(`limit` 0/1/200/201)으로는 이 문제가 안 잡힌다. `delete_all=True`는 **문법적으로 완전히 올바른 입력**이라 Pydantic 검증을 그냥 통과하기 때문이다.
→ 이 이슈는 "잘못된 입력을 막는" 문제가 아니라 **"올바른 입력이지만 비가역적(irreversible)인 행동에 마찰을 두는"** 문제다.

### 7-2. 원인 (코드 근거)

- LLM은 "다 삭제" 요청을 `personal_delete_saved_schedules(delete_all=True)` 로 옳게 번역한다.
- 그 아래 `AppSQLiteStore.delete_all_schedules()`(`fixed/app_store.py`)는 확인 없이 전부 `DELETE` 하는 순수 실행 함수다. → store가 SQL만 담당하는 건 정상.
- **막아야 할 층이 비어 있다**: system prompt에도, tool 본문에도 "되묻기/확인" 로직이 전혀 없다.
- 현재 유일한 안전장치는 `_delete_saved_schedules`의 "조건이 **완전히 비었을 때만** 거부" 규칙뿐이다. `delete_all=True`는 명시적 조건이라 이 규칙을 통과한다.
  → 즉 "실수로 조건 없이 삭제"만 막고, **"명시적이지만 대량·비가역 삭제"는 무방비**다.

### 7-3. 설계안 (3층위 — 병행 권장)

되돌릴 수 없는 작업은 프롬프트 하나로 막지 않는다. LLM은 확률적으로 지시를 무시할 수 있으므로, **코드에서 결정적(deterministic)으로** 한 번 더 잡는다.

| 층위 | 방법 | 성격 |
|---|---|---|
| 프롬프트 | system prompt에 "delete_all·대량 삭제는 실행 전 몇 개가 지워지는지 알리고 확인받아라" 규칙 추가 | 필요하지만 단독으론 불충분(확률적) |
| 코드 — 미리보기/확정 2단계 | delete_all 요청 시 바로 삭제하지 않고 "N개가 삭제됩니다. 진행할까요?"를 먼저 반환, 재확인 후 실행 | 확실 |
| 코드 — `confirmed` 기본 False | `personal_delete_saved_schedules`에 `confirmed: bool = False` 인자 추가. False면 삭제 대상 수만 세서 반환, True여야 실제 삭제 | 결정적·검증 가능(권장 축) |

**결정(잠정):** 프롬프트 + `confirmed` 플래그 병행. 사소한 단건 삭제까지 매번 되묻지 않도록, 가드는 `delete_all` 또는 "대량(예: N건 초과)"에만 건다. (N 임계값은 열린 질문으로 남김.)

### 7-4. 검증 계획

- **가드 발동:** "일정 다 삭제해줘" → 즉시 삭제되지 않고 "N개 삭제 예정, 진행할까요?" 확인 응답이 나오는가. "응/진행해" 재확인 후에만 실제로 사라지는가.
- **가드 우회 안 됨:** 재확인 없이 두 번째 삭제 호출이 바로 실행되지 않는가.
- **정상 흐름 회귀:** 단건 삭제("그 회의 하나 지워줘")는 여전히 마찰 없이 동작하는가.
- **경계:** 저장된 일정 0개일 때 "다 삭제" → 지울 게 없다고 안전하게 답하는가.
