## 과제 목표

이번 과제를 시작하면서, 단순히 기능 3개를 "돌아가게" 만드는 걸 넘어서
아래 세 가지를 내 것으로 만들고 싶었다.

- **AI 에이전트가 도구를 스스로 고르는 구조를 이해하고 싶었다.**
  LLM이 직접 호출하는 LangChain `@tool` 3개(생성·조회·삭제)를 만들면서,
  "Nana가 어떤 상황에 어떤 도구를 고르는가"를 코드 레벨에서 이해하는 것이 목표였다.

- **데이터가 대화별로 분리되는 원리를 익히고 싶었다.**
  하나의 저장소를 여러 대화가 공유할 때 `session_id`로 범위를 나눠,
  다른 사람의 데이터가 새지 않게 하는 방법을 손으로 짜보며 익히고 싶었다.

- **결과가 아니라 "왜 그렇게 동작하는지"까지 책임지고 싶었다.**
  AI에게 코드를 받아 붙이는 대신, 개념을 이해하고 직접 구현해서
  내가 짠 모든 줄을 설명할 수 있는 상태로 과제를 마치는 것이 가장 큰 목표였다.
  (희망 진로인 AI 서비스 품질·안전성과도 맞닿아 있다고 생각했다.)

### 도구 3개 구현 (생성 / 조회 / 삭제)

- AI 활용 내용 :
  코드를 짜달라고 하는 대신, "개념을 예시로 설명받기 → 구현 순서를 내가 먼저 말로 세워 검토받기 → 빈칸 틀(`___`)만 받아 직접 채우기 → 피드백받기" 순서로 진행했다.

- 직접 수정한 부분 :
  세 함수 본문을 전부 직접 작성했다.
  (a) `attendees`가 `None`일 때 `[]`로 바꾸는 방어 코드
  (b) 조회를 `PERSONAL_SCHEDULES` 원본이 아니라 `_current_session_schedules()`(내 대화 범위)에서 시작하도록 한 것
  (c) 삭제 조건을 `id != schedule_id or scope != session_id`(남길 조건)로 뒤집어 작성한 부분을 스스로 판단해 넣었다.

- 수정 이유 :
  코드를 받기만 하면 "왜 이렇게 동작하는지"를 설명할 수 없었고, 직접 구현해야 회고와 면접에서 근거를 가지고 설명할 수 있다고 판단했다.

### 검증 시나리오 직접 설계

- AI 활용 내용 :
  "짰다 ≠ 작동한다"를 확인하려고, 정상 케이스만이 아니라 위험 시나리오를 일부러 만들어 검증하는 테스트를 설계했다.

- 직접 수정한 부분 :
  두 개의 다른 대화(철수·영희)를 흉내 내고, 영희 일정에 철수와 똑같은 `id`를 강제로 박은 뒤 철수가 삭제를 호출해도 영희 일정이 살아남는지 확인하는 케이스를 추가해보았다.

- 수정 이유 :
  ID가 같아도 대화 범위가 다르면 지우면 안 된다는 요구사항이 실제로 지켜지는지는, 충돌 상황을 만들어야만 검증되기 때문이다.

## 구현하면서 고민한 점
막혔던 부분, 고민한 내용, 해결 방법을 자유롭게 적어요.

이번 과제는 코드를 받지 않고, 구현 순서를 먼저 말로 세운 뒤 직접 채우는 방식으로 진행했다. 그 과정에서 내가 틀리게 생각했던 지점과, 그걸 어떻게 교정했는지를 그대로 남긴다.

### 고민 1 — `session_id`를 "만든다"고 생각했던 것
처음 계획에는 "먼저 session_id를 만든다"라고 적었다. 그런데 `session_id`는 내가 새로 만드는 값이 아니라, `current_session_scope()`를 호출하면 "이 대화가 누구 것인지"를 이미 알려주는 값이라는 걸 알게 됐다. 즉 **만드는 게 아니라 가져다 쓰는 것**이었다. 정작 새로 만들어야 하는 건 일정마다 붙는 번호표 `id`(`_new_personal_id()`)였다. 둘 다 "id"가 들어가서 같은 것으로 착각했는데, **읽어오는 값(`session_id`)과 발급하는 값(`id`)**으로 역할을 분리하니 정리됐다.

### 고민 2 — `attendees`를 왜 빈 리스트로 바꾸는가
"attendees가 비어있으면 `[]`로 바꾼다"는 지시가 처음엔 이해되지 않았다. 확인해 보니, 사용자가 "회의 잡아줘"처럼 참석자를 말하지 않으면 `attendees`는 `None`으로 들어온다. 이때 일정 dict에 `"attendees": None`이 박히면, 나중에 "참석자 목록을 돌면서 보여줘" 같은 코드에서 `None`은 순회할 수 없어 에러가 난다. 빈 리스트 `[]`는 "참석자 0명"으로 안전하게 처리된다.

> 출석부에 참석자 칸을 비워두는 것(`[]`)과 칸 자체를 찢는 것(`None`)의 차이라고 이해했다. 빈 칸은 "0명이네" 하고 넘어가지만, 칸이 없으면 멈춰버린다.

이걸 품질 관점으로 다시 보니, **"사용자가 참석자를 안 적는 시나리오"를 미리 막는 방어 코드**라는 걸 알게 됐다. "이렇게 입력하면 망가지지 않을까?"를 떠올리고 미리 대비하는 것이 품질 작업이라는 걸 체감했다.

### 고민 3 — 삭제 조건이 왜 `and`가 아니라 `or`인가
이 함수가 제일 어려웠다. 나는 "ID가 같고(`and`) 내 범위인 것"을 찾아 지우는 줄 알았는데, 코드는 반대로 **"남길 것만 골라 새 목록을 만드는"** 방식이었다. 그래서 조건을 뒤집어야 했다.

- 지울 조건 : `ID 같다 AND 내 범위다`
- 남길 조건(그 반대) : `ID 다르다 OR 내 범위가 아니다`

예시로 확인했다. 철수가 `personal_aaa` 삭제를 요청하고, 저장소에 같은 ID를 가진 영희의 일정이 있다고 하면 —

1. 철수의 `personal_aaa` : ID 같고 + 내 범위 → 남길 이유 없음 → 지워짐
2. 영희의 `personal_aaa` : ID는 같지만 **내 범위가 아님** → `or` 조건 하나 충족 → **남겨짐**

`or` 덕분에 영희 일정이 살아남았다. 만약 여기서 `and`로 쓰거나 ID만 비교했다면, 영희 일정까지 지워지는 **데이터 손실 사고**가 났을 것이다. "삭제 함수의 권한 범위 검증"이 품질·안전성에서 가장 무겁게 봐야 할 항목이라는 걸 이 부분에서 배웠다.

### 고민 4 — 왜 `PERSONAL_SCHEDULES =`가 아니라 `PERSONAL_SCHEDULES[:] =` 인가
- `PERSONAL_SCHEDULES = new_list` : 변수 이름만 새 리스트로 갈아끼움
- `PERSONAL_SCHEDULES[:] = new_list` : 기존 리스트 객체는 두고 내용물만 통째로 교체

다른 코드가 이미 원본 리스트 객체를 가리키고 있을 수 있어서, `=`로 갈아끼우면 나만 새 리스트를 보고 남들은 옛날 것을 보게 된다. `[:] =`로 내용만 바꿔야 모두가 같은 최신 상태를 본다. (새 화이트보드를 사 오는 것 vs 기존 보드를 지우고 다시 쓰는 것의 차이로 이해했다.)

## 과제 회고 (KPT)

- **Keep** (좋았고 계속 유지할 점) : 코드를 받지 않고 "계획 → 직접 구현 → 검증"으로 진행해, 모든 줄을 설명할 수 있게 된 점. "작동하나?"를 정상 케이스가 아니라 충돌·프라이버시 같은 위험 시나리오로 확인한 점.

- **Problem** (아쉬웠거나 막혔던 점) : `id`/`session_id` 혼동, 삭제 조건 `and`/`or` 같은 부분에서 멈췄다. 자료구조와 조건 뒤집기(드모르간) 같은 기본기가 더 탄탄했다면 덜 막혔을 것이다. 이번에 약한 지점을 정확히 알게 된 건 오히려 수확이다.

- **Try** (다음에 시도해볼 점) : 입력 검증을 어디까지 도구가 막고 어디부터 프롬프트로 막을지 기준을 세워보기. 시스템 프롬프트를 바꿔가며 Nana가 올바른 도구를 고르는 비율을 비교하는 평가를 직접 돌려보기.
